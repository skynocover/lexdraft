## Context

書狀 pipeline 有三條產出段落的 track：

1. **Template renderer** (`templateRenderer.ts`) — 從範本渲染固定格式段落，已有 `isAutoGenerated()` 跳過證據方法
2. **AI writer loop** (`briefPipeline.ts` L358-390) — 遍歷 `store.sections`，每個 section 呼叫 Claude/Gemini 寫段落
3. **Evidence formatter** (`evidenceFormatter.ts`) — 從 exhibits 表程式化產出證據清單

Track 2 沒有任何機制排除系統管理的 section，導致 AI 可能規劃並寫出「證據方法」，與 Track 3 重複。

## Goals / Non-Goals

**Goals:**
- 在 writer loop 之前建立「系統管理 section」的邊界檢查，阻止 AI writer 處理這些 section
- 統一 `AUTO_HEADINGS` 常數定義，避免多處重複
- Prompt 層輔助提示 AI 不要規劃證據方法

**Non-Goals:**
- 不統一證據方法的 section 命名（中文編號前綴），這需要 template 結構耦合
- 不修改 `evidenceFormatter` 的產出格式
- 不新增測試框架

## Decisions

### 1. 過濾位置：writer loop 之前（而非組裝時）

在 `briefPipeline.ts` L358 的 `for` loop 之前過濾 `store.sections`。

**替代方案**：在組裝 `allParagraphs` 時過濾（doc 原方案）
**否決原因**：組裝時過濾意味著 AI writer 已經為該 section 呼叫了 API，浪費 token 和時間。提前過濾從源頭阻止。

### 2. 共用常數放在 `strategyConstants.ts`

將 `AUTO_HEADINGS` 從 `templateRenderer.ts` 移至 `strategyConstants.ts`，因為：
- `strategyConstants.ts` 已是 pipeline 共用常數的集中地
- `templateRenderer.ts` 和 `briefPipeline.ts` 都 import 它

**替代方案**：新建 `pipelineConstants.ts`
**否決原因**：過度拆分，現有 `strategyConstants.ts` 已承擔此角色。

### 3. Prompt 層為輔助，不作為唯一防線

在 `SECTION_RULES` 加一句「不要規劃證據方法」，但不依賴 AI 遵守。Code 層過濾才是硬保證。

## Risks / Trade-offs

- **[誤濾正常段落]** → `AUTO_HEADINGS` 匹配用 `includes()`，若 AI 在某段的 `section` 欄位裡意外包含「證據」兩字可能被誤濾。風險極低：section 名稱遵循「壹、前言」「參、事實及理由」格式，不會含「證據方法」或「證據」子字串。
- **[未來新增系統管理 section]** → 新增時需更新 `AUTO_HEADINGS` 陣列。目前只有證據方法，複雜度可控。
